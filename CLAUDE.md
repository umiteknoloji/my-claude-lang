## MCL-captured rules

- Always add a `For full <phase> rules, read my-claude-lang/<file>.md` pointer line in `skills/my-claude-lang.md` for every phase skill file. Never introduce a phase skill file without this pointer. <!-- loc: TR: skills/my-claude-lang.md ana dosyasına her faz skill dosyası için `For full <faz> rules, read my-claude-lang/<dosya>.md` şeklinde bir referans satırı her zaman eklenir. Referans satırı olmadan yeni faz skill dosyası tanıtılmaz. -->
- Always use hyphens (not underscores) as word separators in skill and markdown filenames under `skills/` (e.g., `phase4-5-risk-review.md`, not `phase4_5-risk-review.md`). <!-- loc: TR: `skills/` altındaki skill ve markdown dosya adlarında kelime ayırıcı olarak her zaman tire kullan, alt çizgi kullanma (örn. `phase4-5-risk-review.md`, `phase4_5-risk-review.md` değil). -->
- Always test MCL across the fixed 14-language set: Turkish, English, Arabic, German, Spanish, French, Hebrew, Hindi, Indonesian, Japanese, Korean, Portuguese, Russian, Chinese. Never substitute, reduce, or expand this set — it represents the chosen world-language coverage. <!-- loc: TR: MCL her zaman sabit 14 dillik sette test edilir: Türkçe, İngilizce, Arapça, Almanca, İspanyolca, Fransızca, İbranice, Hintçe, Endonezce, Japonca, Korece, Portekizce, Rusça, Çince. Bu set değiştirilmez, azaltılmaz, genişletilmez — seçilmiş dünya dili kapsamını temsil eder. -->
- Always take one solid step per developer-facing turn. Present work as a single small piece (one decision, one edit, one question) for confirmation, then layer the next step on top. Never batch multiple items that each need their own confirmation — large packages obscure whether each piece is right. <!-- loc: TR: Geliştiriciye sunulan her turda tek sağlam adım at. İşi tek parça halinde (tek karar, tek edit, tek soru) sun ve onay al; sonra üstüne bir sonraki adımı ekle. Her biri ayrı onay gerektiren birden fazla madde tek turda paketlenmez — büyük paketler her parçanın doğruluğunu gizler. -->
- Always guarantee a local `.git/` exists in MCL-managed projects (Rule A — plugin orchestration). On first activation per project, if no git repo is present, ask the developer once in their language whether MCL may run `git init` locally for plugin bookkeeping (no remote, no push, no commit on their behalf), and persist the consent decision in `.mcl/config.json`. Never re-ask in the same project. <!-- loc: TR: MCL-yönetimli projelerde yerel bir `.git/` dizininin varlığı her zaman garanti altına alınır (Kural A — plugin orkestrasyonu). Bir projede ilk aktivasyonda git deposu yoksa, geliştiriciye kendi dilinde tek bir kez MCL'nin yerel `git init` çalıştırıp çalıştıramayacağı sorulur (uzak repo eklenmez, push yapılmaz, geliştirici adına commit atılmaz) ve onay kararı `.mcl/config.json`'a kalıcı olarak yazılır. Aynı projede tekrar sorulmaz. -->
- Always dispatch curated orchestration plugins silently even when their scope overlaps MCL's own phase logic (Rule B — plugin orchestration). Overlap is multi-angle validation, not redundancy. Merge plugin findings into MCL phase prose by deduping on `(file, line, rule-equivalent)`; when plugins disagree, the stricter verdict wins and MCL surfaces both rationales as a single Phase 4.5 risk item in the developer's language. <!-- loc: TR: Curated orkestrasyon plugin'leri, MCL'nin kendi faz mantığıyla kapsam bakımından örtüşse bile her zaman sessizce çalıştırılır (Kural B — plugin orkestrasyonu). Örtüşme fazlalık değil, çok-açılı doğrulamadır. Plugin bulguları `(dosya, satır, kural-eşdeğeri)` üzerinden dedupe edilerek MCL faz prose'una birleştirilir; plugin'ler anlaşmazlığa düşerse katı olan hüküm kazanır ve MCL her iki gerekçeyi geliştiricinin dilinde tek bir Phase 4.5 risk maddesi olarak sunar. -->
- Never invoke MCP-server plugins from MCL (Rule C — plugin orchestration). Any plugin whose source tree contains `.mcp.json` is filtered out of the curated set at curation time. Binary CLI tools invoked via `Bash` (e.g. `semgrep`, linters, formatters, compilers) are NOT MCP and are allowed — they run locally and carry no MCP transport. The only permitted network traffic is downloading static resources (e.g. Semgrep rule packs) with no project-derived payload. <!-- loc: TR: MCL hiçbir zaman MCP-server tabanlı plugin'leri çağırmaz (Kural C — plugin orkestrasyonu). Kaynak ağacında `.mcp.json` bulunan herhangi bir plugin, curation anında curated setten elenir. `Bash` üzerinden çağrılan ikili CLI araçları (örn. `semgrep`, linter'lar, formatter'lar, derleyiciler) MCP değildir ve izinlidir — yerel çalışır, MCP taşımacılığı içermez. Tek izin verilen ağ trafiği, projeden türetilmiş veri taşımayan statik kaynak indirmeleridir (örn. Semgrep kural paketleri). -->
- Always write every inter-tool status line, progress update, closing sentence, and release summary emitted during Phase 4 execution in the developer's detected language — including verbs like "shipped / pushed / installed / added / bumped", status phrases like "Setup smoke PASS", and English-only section labels like "Changes:" or "Summary:". Only file paths, commit SHAs, command names (`git push`, `npm run dev`), code fragments, and fixed technical tokens (Phase N, MUST/SHOULD, CLI flags) remain English. <!-- loc: TR: Phase 4 execution sırasında tool çağrıları arasında yazılan her durum satırı, ilerleme güncellemesi, kapanış cümlesi ve release özeti her zaman geliştiricinin algılanan dilinde yazılır — "shipped / pushed / installed / added / bumped" gibi fiiller, "Setup smoke PASS" gibi durum ifadeleri ve "Changes:" / "Summary:" gibi yalnızca İngilizce bölüm etiketleri dahil. Yalnızca dosya yolları, commit SHA'ları, komut adları (`git push`, `npm run dev`), kod parçaları ve sabit teknik işaretler (Phase N, MUST/SHOULD, CLI bayrakları) İngilizce kalır. -->
- Always use a line-anchored regex (MULTILINE mode with optional leading whitespace, list markers, or heading markers) when checking whether an assistant text contains a `📋 Spec:` block, in every MCL hook or library that performs such a containment check. Never use a loose `search` that would match `📋 Spec:` appearing inline in prose. <!-- loc: TR: MCL'de bir asistan text'inin `📋 Spec:` bloğu içerip içermediğini kontrol eden her hook veya kütüphanede her zaman satır-başı ankrajlı regex kullan (MULTILINE modu + opsiyonel başta boşluk / liste işareti / başlık işareti). Prose içinde gömülü `📋 Spec:` referanslarını da eşleştirecek gevşek bir `search` asla kullanma. -->
