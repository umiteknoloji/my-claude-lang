## Devtime Plan Critique (subagent)

When producing a plan during an MCL development session (devtime — you + Ümit), automatically launch a `general-purpose` subagent (model: `claude-sonnet-4-6`, extended thinking enabled) with a plan-critique prompt to evaluate the plan before presenting it. Using a different model than the main session (Opus 4.7) provides genuine second-opinion bias diversity; Sonnet 4.6 with extended thinking is ideal for critique reasoning at ~5x lower cost. (`pr-review-toolkit:code-reviewer` is scoped to code artifacts — wrong tool for plan text.) The subagent applies ten lenses:

1. **Root cause depth** — was the root cause genuinely found, or did the plan stay at the surface symptom?
2. **Side-effect estimation** — were downstream effects estimated (Phase 4 impact-lens logic: imports, shared state, contract breakage, schema effects)?
3. **Null/empty cases** — did the plan consider "what if this is empty / missing / never triggered"?
4. **Disambiguation completeness** — are there unanswered questions that would change the approach? Any hidden assumption?
5. **Plan ambiguity** — is any step underspecified, or does any step assume context that was not confirmed?
6. **Hallucination check** — are factual claims in the plan verified? Any "X tool exists" or "Y behaves like Z" assertions that were not confirmed against the actual codebase or docs?
7. **Conflict with existing features** — does the planned feature overlap with something already in FEATURES.md? Would two parts of the system do the same job?
8. **Extension over modification** — does the plan extend existing code or modify it? Modification without necessity violates the CLAUDE.md rule.
9. **User visibility** — is the feature technically correct but invisible to the user? Avoid "it exists but nobody sees it" features unless the value is explicit.
10. **Testability** — is the plan deterministically testable, or only behavioral? If behavioral, is there a skip-detection control (lens, audit entry, post-check)?

Present the critique alongside the plan. If Ümit rejects the critique, discard it and proceed with the original plan. If accepted, incorporate the improvements before execution.

**Scope: devtime only.** This rule does NOT apply during runtime (user project sessions). Ümit may skip critique at any point — no enforcement. <!-- loc: TR: Devtime plan critique: general-purpose subagent (Sonnet 4.6, extended thinking), 10 lens, Ümit kabul/red eder. Sadece devtime. -->

## Root Cause Discipline

Full specification: `docs/root-cause-discipline.en.md` (TR: `docs/root-cause-discipline.md`)

- **Devtime (sen + CC için):** "Bu neden böyle?" sorusunu tekrarla — cevap gelince tekrar sor. Her halkada 3 check sıralı: (1) görünür süreç, (2) removal test — sebep kalkarsa üst halka çözülür mü, (3) falsification — sebep doğruysa kontrol edilebilir X gözlemlenmeli. Biri fail ederse geri dön. Müdahale noktası: kök sebep değil, sıfır yan etki üreten halka. Kök sebep + çözüm birlikte raporlanır. Tıkanırsa şeffaflık — halüsine etme. <!-- loc: TR: 3 check sıralı, sıfır yan etki halkası, kök sebep + çözüm birlikte, tıkanırsa şeffaflık. -->
- **Runtime (MCL kullanıcıları için — Root Cause Chain):** Phase 4'te ilk sebep bulununca dur değil — 3 check sıralı uygula (görünür süreç → removal test → falsification), geçerse derine in, fail ederse geri dön. Müdahale noktası: kök sebebe en yakın, sıfır yan etki üreten halka. Risk item: kök sebep + önerilen çözüm + çözüm doğrulama sonucu birlikte. Anti-sycophancy gibi temel: atlanması yasak, Phase 4 tamamlanmış sayılmaz. <!-- loc: TR: Phase 4 Root Cause Chain: 3 check sıralı, sıfır yan etki halkası, kök sebep + çözüm + doğrulama. Atlanması yasak. -->

- Always keep every documentation and reference file in the project up to date. When a feature is added, changed, or removed, update FEATURES.md, README.md, README.tr.md, and any affected docs/ files in the same commit. No doc file may reference a version older than the current VERSION file. <!-- loc: TR: Projedeki her dokümantasyon ve referans dosyasını her zaman güncel tut. Bir özellik eklendiğinde, değiştiğinde veya kaldırıldığında FEATURES.md, README.md, README.tr.md ve etkilenen docs/ dosyalarını aynı commit'te güncelle. Hiçbir doc dosyası mevcut VERSION dosyasından eski bir sürüme referans veremez. -->

- Always include README.md and README.tr.md updates in every MCL release commit, AND keep them version-independent. A version bump (`VERSION` change + tagged release commit) is **not accepted** without README updates that reflect any new feature, keyword, env var, or state field introduced in that release. **README.md and README.tr.md MUST NOT contain version numbers anywhere** — no version badges, no `(8.x.0)` annotations next to feature headings, no "since 8.y.0" qualifiers in feature prose. CHANGELOG.md is the single source of truth for version history; README describes the feature, not the release that added it. Each release MUST only **add new feature descriptions** — never re-state version numbers in README. This is the Phase 5 → Phase 6 enforcement point for documentation: **every release plan must list `step N: README.md + README.tr.md update (no version mentions)` as a mandatory step**, and Phase 6 (a) audit-trail check should treat both absence and version-leak as documentation gaps. <!-- loc: TR: Her MCL release commit'inde README.md ve README.tr.md güncellemesi her zaman dahil edilir VE bunlar sürüm-bağımsız tutulur. Versiyon bump'ı (`VERSION` değişikliği + tagged release commit), o release'te tanıtılan yeni özellik / keyword / env var / state field'ı yansıtan README güncellemeleri olmadan **kabul edilmez**. **README.md ve README.tr.md hiçbir yerde sürüm numarası içermemelidir** — sürüm badge'i yok, özellik başlığı yanında `(8.x.0)` ek açıklaması yok, özellik prose'unda "8.y.0'dan beri" niteleyicisi yok. CHANGELOG.md sürüm geçmişi için tek doğru kaynaktır; README, özelliği anlatır, ekleyen release'i değil. Her release SADECE **yeni özellik açıklaması ekleyebilir** — README'de sürüm numarası tekrar belirtilmez. Bu, dokümantasyon için Phase 5 → Phase 6 enforcement noktasıdır: **her release plan'ı `step N: README.md + README.tr.md update (sürüm bahsi yok)` adımını zorunlu adım olarak listelemelidir**, ve Phase 6 (a) audit-trail kontrolü hem eksikliği hem sürüm-sızıntısını dokümantasyon gap'i olarak işlemelidir. -->
