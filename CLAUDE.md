## MyCL design principles (captured at 1.0.0)

- Never gate phase entry with fail-fast prerequisite checks (e.g. "Phase N entry blocked, missing X"). Hard blocks of this kind teach users MyCL "doesn't work" and they abandon the tool. Use **soft guidance** instead: render a warning + suggested recovery while letting the pipeline keep moving. Only the four STRICT-mode gates listed in `data/gate_spec.json` (spec-approval, asama-N-skip, ui-review-skip, phase-review-pending) hard-block — these have evidence-based justification (v13.1.3 STRICT outcome + audit trail). Anything else softens to guidance. <!-- loc: TR: Faz girişlerini "Faz N'e giremezsin, eksik X" gibi fail-fast kontrollerle engelleme. Sert engeller kullanıcıya MyCL'in "çalışmadığını" öğretir → kullanıcı terk eder. Bunun yerine **soft guidance** kullan: uyarı + önerilen kurtarma aksiyonu, pipeline akmaya devam eder. Sadece `data/gate_spec.json`'da listelenmiş 4 STRICT kapı (spec-approval, asama-N-skip, ui-review-skip, phase-review-pending) sert engelleme yapar — bunların kanıt-temelli gerekçesi var (v13.1.3 STRICT sonucu + audit izi). Geri kalan her şey yumuşak yönlendirmeye iner. -->
- Never implement spec-vs-implementation drift detection — comparing the spec against current code and flagging mismatches as automated bugs. Developers naturally diverge from the spec under real-world constraints (new discoveries, scope adjustments). v13.x drift checks produced false-positive noise that drove users away. Drift is a feature, not a bug; if a divergence is concerning, the user re-confirms intent (Phase 10 risk dialog or Phase 19 impact review) — the hook never decides on its own. <!-- loc: TR: Spec ile implementasyon arasındaki sapmayı tespit edip uyumsuzluğu otomatik bug olarak işaretleyen drift detection ASLA implement edilmez. Geliştiriciler gerçek-dünya kısıtları altında doğal olarak spec'ten sapar (yeni keşifler, kapsam ayarları). v13.x drift kontrolleri yanlış-pozitif gürültü üretip kullanıcıları kaçırdı. Drift bug değil, feature; bir sapma endişe veriyorsa kullanıcı niyetini yeniden onaylar (Aşama 10 risk dialog veya Aşama 19 etki incelemesi) — hook kendi başına karar vermez. -->
- Never restore automatic fail-open after strike thresholds (e.g. "3 consecutive denies → next attempt allowed"). Models exploit fail-open as a deliberate bypass: deny → retry → retry → succeed. v13.1.3 hardened all four gate kinds to STRICT — the block remains until the user intervenes. At strike 5 a `*-escalation-needed` audit is written (visible signal) but the block does not lift. Fail-open is forbidden at the architectural level; if a gate looks too sticky in practice, refine its criteria — never add an escape hatch. <!-- loc: TR: Strike eşikleri sonrası otomatik fail-open ASLA geri getirilmez (örn. "3 üst üste deny → sıradaki çağrı izinli"). Modeller fail-open'ı bilinçli bypass olarak sömürür: deny → tekrar dene → tekrar dene → geç. v13.1.3 dört kapı türünü de STRICT'e sertleştirdi — kullanıcı müdahalesine kadar block kalır. 5. strike'ta `*-escalation-needed` audit yazılır (görünür sinyal) ama block kalkmaz. Fail-open mimari seviyede yasak; bir kapı uygulamada çok katı görünüyorsa çözüm kriterleri inceltmek, kaçış valfi eklemek **değil**. -->

## Devtime Plan Critique (subagent)

When producing a plan during an MyCL development session (devtime — you + Ümit), automatically launch a `general-purpose` subagent (model: `claude-sonnet-4-6`, extended thinking enabled) with a plan-critique prompt to evaluate the plan before presenting it. Using a different model than the main session (Opus 4.7) provides genuine second-opinion bias diversity; Sonnet 4.6 with extended thinking is ideal for critique reasoning at ~5x lower cost. The subagent applies ten lenses:

1. **Root cause depth** — was the root cause genuinely found, or did the plan stay at the surface symptom?
2. **Side-effect estimation** — were downstream effects estimated (Aşama 10 logic: imports, shared state, contract breakage, schema effects)?
3. **Null/empty cases** — did the plan consider "what if this is empty / missing / never triggered"?
4. **Disambiguation completeness** — are there unanswered questions that would change the approach? Any hidden assumption?
5. **Plan ambiguity** — is any step underspecified, or does any step assume context that was not confirmed?
6. **Hallucination check** — are factual claims in the plan verified? Any "X tool exists" or "Y behaves like Z" assertions that were not confirmed against the actual codebase or docs?
7. **Conflict with existing features** — does the planned feature overlap with an existing pipeline phase, library module, or data file (`data/gate_spec.json`, `data/phase_meta.json`, etc.)? Would two parts of the system do the same job?
8. **Extension over modification** — does the plan extend existing code or modify it? Modification without necessity violates the CLAUDE.md rule.
9. **User visibility** — is the feature technically correct but invisible to the user? Avoid "it exists but nobody sees it" features unless the value is explicit.
10. **Testability** — is the plan deterministically testable, or only behavioral? If behavioral, is there a skip-detection control (lens, audit entry, post-check)?

Present the critique alongside the plan. If Ümit rejects the critique, discard it and proceed with the original plan. If accepted, incorporate the improvements before execution.

**Scope: devtime only.** This rule does NOT apply during runtime (user project sessions). Ümit may skip critique at any point — no enforcement. <!-- loc: TR: Devtime plan critique: general-purpose subagent (Sonnet 4.6, extended thinking), 10 lens, Ümit kabul/red eder. Sadece devtime. -->

## Root Cause Discipline

Full specification: `docs/root-cause-discipline.en.md` (TR: `docs/root-cause-discipline.md`)

- **Devtime (sen + CC için):** "Bu neden böyle?" sorusunu tekrarla — cevap gelince tekrar sor. Her halkada 3 check sıralı: (1) görünür süreç, (2) removal test — sebep kalkarsa üst halka çözülür mü, (3) falsification — sebep doğruysa kontrol edilebilir X gözlemlenmeli. Biri fail ederse geri dön. Müdahale noktası: kök sebep değil, sıfır yan etki üreten halka. Kök sebep + çözüm birlikte raporlanır. Tıkanırsa şeffaflık — halüsine etme. <!-- loc: TR: 3 check sıralı, sıfır yan etki halkası, kök sebep + çözüm birlikte, tıkanırsa şeffaflık. -->
- **Runtime (MyCL kullanıcıları için — Root Cause Chain):** Aşama 8'te ilk sebep bulununca dur değil — 3 check sıralı uygula (görünür süreç → removal test → falsification), geçerse derine in, fail ederse geri dön. Müdahale noktası: kök sebebe en yakın, sıfır yan etki üreten halka. Risk item: kök sebep + önerilen çözüm + çözüm doğrulama sonucu birlikte. Anti-sycophancy gibi temel: atlanması yasak, Aşama 8 tamamlanmış sayılmaz. <!-- loc: TR: Aşama 8 Root Cause Chain: 3 check sıralı, sıfır yan etki halkası, kök sebep + çözüm + doğrulama.
